# Requirements: PhishGuard

**Defined:** 2026-06-08
**Core Value:** Accurate, explainable phishing detection achieved through a robust distributed architecture and strict cybersecurity practices without the overhead of heavy SPA frameworks or paid APIs.

## v1 Requirements

### Authentication & Security

- [ ] **SEC-01**: User can log in and receive a JWT session token
- [ ] **SEC-02**: Passwords are hashed using bcrypt upon creation
- [ ] **SEC-03**: System denies access to protected routes without a valid JWT
- [ ] **SEC-04**: System supports Admin and Analyst roles, blocking Analyst from Admin routes
- [ ] **SEC-05**: System logs audit events (logins, submissions) in structured JSON format

### Interface (UI)

- [ ] **UI-01**: User can access a Dashboard styled with Tailwind CSS (Dark Mode)
- [ ] **UI-02**: User can submit an email text or metadata via a submission form
- [ ] **UI-03**: User can view a history table of past analyzed emails (Low/Medium/High score)
- [ ] **UI-04**: User can logout of the dashboard

### API Gateway

- [ ] **GW-01**: Gateway accepts JSON payloads and validates them with Pydantic
- [ ] **GW-02**: Gateway forwards analysis requests to the AnalysisService via gRPC
- [ ] **GW-03**: Gateway handles gRPC timeouts or unavailability gracefully (503 response)

### Analysis Service

- [ ] **ANL-01**: Service runs as an independent process listening on gRPC port 50051
- [ ] **ANL-02**: Service applies heuristic scoring (Regex, keywords) to detect urgency, bad domains, etc.
- [ ] **ANL-03**: Service returns a cumulative score (Low/Medium/High) and a short textual justification

## Out of Scope

| Feature | Reason |
|---------|--------|
| Direct connection to Gmail/Outlook | Excluded to fit the 15-day timeframe |
| Complex `.eml` file parsing | Users will submit text/metadata directly |
| Full AI/ML model for analysis | Rules/heuristics are sufficient and meet the constraints |
| Separate frontend SPA (React/Vue) | Too much build overhead and CORS complexity; Jinja templates are sufficient |
